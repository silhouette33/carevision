# CareVision 프로젝트 정리

> AI 기반 독거노인 스마트 케어 모니터링 시스템 (캡스톤 디자인 프로젝트) 의 기획·요구사항·설계·구현·현황을 정리한 마스터 문서.

---

## 1. 프로젝트 개요

### 1.1 한 줄 소개
**CareVision** — 카메라 영상과 AI를 활용해 독거노인의 낙상·복약 위험을 실시간으로 감지하고, 보호자에게 즉시 알림을 보내는 스마트 케어 모니터링 시스템.

### 1.2 프로젝트 배경
- **컴퓨터공학과 캡스톤 디자인** 프로젝트 (6인 팀)
- 한국은 2025년을 기점으로 초고령 사회 진입, 독거노인 비중이 빠르게 증가
- 고령층 사고의 핵심 리스크 두 가지:
  1. **낙상** — 골절·장기 와상 → 사망으로 이어지는 비율 높음
  2. **복약 누락** — 만성질환 관리 실패, 복합 부작용 위험
- 기존 시니어 케어 서비스는 대부분 "버튼을 눌러야 호출"되는 수동형 → 의식을 잃은 상태에서는 무용지물
- **AI 영상 분석으로 "수동 → 자동"** 전환이 목표

### 1.3 해결하고자 하는 문제
| 문제 | 기존 접근 | CareVision 접근 |
|------|----------|----------------|
| 낙상 발견 지연 | 수동 버튼 / 이웃 발견 | 카메라 + AI 포즈 분석, 3프레임 연속 판정 |
| 복약 누락 파악 불가 | 보호자가 매일 전화 확인 | 스케줄 + 카메라 객체 감지로 자동 체크 |
| 프라이버시 부담 | 24시간 상시 영상 공유 | 평상시 스트림 비활성, 위급 시에만 자동 활성화 |
| 알림 지연 | 기기 안에서만 알림 | FCM 푸시로 보호자 스마트폰 즉시 전송 |

## 2. 요구사항

### 2.1 기능 요구사항
1. **복약 스케줄 관리** — 보호자가 피보호자의 복약 일정 등록/수정/삭제 (시간대·요일별)
2. **복약 감지** — 카메라 영상에서 알약/캡슐을 감지하고 스케줄 시간 기준으로 복용 여부 판정 (완료 / 의심 / 누락)
3. **낙상 감지** — MediaPipe Pose로 관절 좌표를 추출해 머리-골반 위치 관계 기반 낙상 판정, 연속 프레임 분석으로 오탐 감소
4. **긴급 알림** — 위급 확정 시 보호자 앱에 FCM 푸시 전송
5. **실시간 스트리밍** — 평상시 비활성, 위급 알림 수신 시 보호자 앱에서 카메라 영상 확인 가능
6. **환자·알림 관리** — 보호자 1명이 여러 피보호자 관리, 알림 이력 열람/읽음 처리

### 2.2 비기능 요구사항
- **정확도**: 단일 프레임이 아닌 **연속 3프레임 이상** 근거 기반으로 판정해 오탐지 최소화
- **프라이버시**: 상시 스트리밍 금지, 이벤트 기반 활성화
- **골든타임**: 감지 → 알림까지 지연 1초 이내 목표
- **모바일 우선**: 보호자 UI는 Capacitor 기반 앱화를 전제로 모바일 레이아웃 설계
- **확장성**: AI/백엔드/프론트 분리 아키텍처, 라즈베리파이 엣지 배포 가능한 경량 구조

## 3. 서비스 시나리오 (최종 결과)

### 3.1 사용자 플로우

```
[피보호자 집]
  카메라 → AI 서버가 매 프레임 분석
    ├─ 복약 시간대: 알약/캡슐 객체 감지 → 복약 여부 판정
    └─ 상시: 포즈 분석 → 낙상 감지
         │
         └─ 판정 결과를 백엔드에 전송

[백엔드]
  이벤트 수신 → DB 기록 → 보호자 FCM 푸시

[보호자 앱]
  ├─ 평상시: 환자 목록, 복약 진행률, 알림 이력 열람
  └─ 위급 알림 수신:
       └─ 탭 → 카메라 스트림 자동 활성화 → 영상 확인
```

### 3.2 화면 구성 (구현 완료)
| 화면 | 핵심 내용 |
|------|----------|
| 로그인 | 보호자 계정 로그인/회원가입, 모바일 레이아웃 |
| 대시보드 | 담당 환자 카드 목록 (이름/나이/주소/오늘의 복약 진행률) |
| 환자 상세 | 복약 스케줄 등록·목록, 복약 현황 (날짜별) |
| 긴급 (카메라) | 환자 선택 후 실시간 스트림 + AI 감지 오버레이 (bbox, 낙상 경고) |
| 알림 | 낙상/복약 미이행 알림 목록, 읽음 처리 |
| 마이 | 담당 환자 수·주간 복약률·미확인 알림 통계, 알림 권한 설정, 계정 설정 |

### 3.3 핵심 시연 흐름 (시연회용)
1. 로그인 → 대시보드에서 "김순자 님" 선택
2. 복약 스케줄 등록 (08:00 혈압약)
3. 카메라 앞에서 알약 들기 → AI 서버가 감지 → 화면에 bbox 표시, 체크 처리
4. 낙상 동작 시뮬레이션 → 3프레임 누적 판정 → 긴급 알림 토스트 + FCM 푸시
5. 알림 탭 → 카메라 화면 자동 진입 → 실시간 영상 확인

## 5. 기술 스택과 선택 이유

### 5.1 AI 서버 — Python 3.11 · FastAPI · YOLOv8 · MediaPipe · ONNX Runtime

#### 왜 이 조합인가
- **FastAPI**: 비동기 기반 고성능 웹프레임워크. AI 추론처럼 I/O와 GPU 호출이 섞이는 워크로드에 적합. Swagger 자동 생성으로 다른 팀원이 API 테스트하기 쉬움
- **YOLOv8 (Ultralytics)**: 객체 감지 SOTA급 성능 + `ultralytics` 한 줄 API로 학습/추론 모두 가능 → 학생 프로젝트에서 생산성 최고
- **MediaPipe Pose**: 구글의 경량 포즈 추정기, 33개 관절 좌표를 CPU에서도 30fps 수준으로 추출 가능. 낙상 감지에 별도 모델 학습 없이도 기본 판정 가능
- **ONNX Runtime**: 사전학습 모델(`pills_detection.onnx`)을 프레임워크 독립적으로 로드 가능 → PyTorch/TensorFlow 혼재 환경 회피

#### 어떻게 개발했는가
- `ai/main.py` FastAPI 진입점 + `ai/api/routes.py` 라우터 분리
- 모델별 파이프라인 클래스화 (`pipelines/fall_detector.py`, `pipelines/medication_detector.py`)
- 연속 프레임 상태는 카메라 ID 단위로 in-memory dict에 누적
- 복약 모델은 **3단계 폴백 체인**: `medication.pt` (파인튜닝) → `pills_detection.onnx` (일반) → `yolov8n.pt` (COCO bottle/cup)

### 5.2 백엔드 — Node.js 20 · Express · PostgreSQL 18 · Prisma

#### 왜 이 조합인가
- **Node/Express**: 프론트(React)와 언어 일치 → JSON 구조·타입·유틸 공유 쉬움
- **Prisma**: 타입 안전 ORM, `schema.prisma` 한 파일로 DB 구조 정의 → 마이그레이션 자동 생성, 팀원 간 스키마 합의 비용 최소
- **PostgreSQL**: 프로젝트 엔티티가 관계형 (User → Patient → Medication/DetectionLog/Notification)으로 정규화가 자연스러움, FCM 토큰·JSON 필드도 JSONB로 무난히 수용

#### 어떻게 개발했는가
- 라우트/컨트롤러/미들웨어/서비스 4계층 분리
- JWT 인증 미들웨어 (`middleware/auth.js`)를 모든 보호 라우트에 선부착
- AI 서버 호출은 `services/ai.js`로 추상화해 라우트에서 분리
- AI → 백엔드 방향 호출용 `backend_client.py`도 AI 쪽에 대칭 배치 (양방향 연동)

### 5.3 프론트엔드 — React 18 · Vite · Tailwind CSS · lucide-react

#### 왜 이 조합인가
- **Vite**: HMR이 webpack 대비 월등히 빠름, React + Tailwind 조합 초기 세팅이 5분 이내
- **Tailwind**: 유틸리티 클래스 방식으로 디자인 시스템 없이도 일관된 스타일 빠르게 구현, 모바일 반응형 (`max-w-[480px]`)을 선언적으로 처리
- **lucide-react**: 이모지 아이콘 → SVG 아이콘으로 교체해 모바일에서도 선명, `strokeWidth` 동적 변경으로 활성 탭 강조
- **Capacitor 전제**: 동일한 React 코드를 모바일 앱으로 번들 가능 → 별도 네이티브 앱 개발 불필요

#### 어떻게 개발했는가
- 페이지별 파일 분리 (`pages/DashboardPage.jsx` 등), 라우팅은 단순 `useState` 기반
- **목업 API 패턴**: `api/client.js`의 `useMock()` 래퍼로 실제 API 실패 시 로컬 목 데이터 반환 → DB/백엔드 없이도 프론트 개발·시연 가능
- 공통 레이아웃: `min-h-screen bg-slate-100 max-w-[480px] mx-auto font-sans` 패턴을 각 페이지 루트에 적용

### 5.4 인프라 — Docker · GitHub Actions · Firebase FCM

#### 왜 이 조합인가
- **Docker Compose**: AI(Python) + 백엔드(Node) + DB(Postgres) 3종을 한 번에 띄워 라즈베리파이 배포 단순화
- **GitHub Actions**: 프론트 GitHub Pages 자동 배포, PR 체크용 경량 CI
- **Firebase FCM**: 안드로이드/iOS 동시 지원 푸시, 무료 할당량으로 학생 프로젝트에 충분

## 6. 핵심 기술 상세

### 6.1 낙상 감지 — 휴리스틱에서 LSTM으로

#### 1단계: MediaPipe Pose + 규칙 기반 (운영 중)
- 입력: 프레임당 33개 관절 좌표 (x, y, z, visibility)
- 판정 로직:
  - **주조건**: 머리(Nose) y > 골반(Hip) y (머리가 골반보다 아래)
  - **부조건**: 어깨-골반 y 차이 < 0.15 (몸이 수평)
- **연속 3프레임** 누적 시 emergency 확정 → 오탐 방지
- 장점: 별도 학습 불필요, CPU로도 실시간
- 한계: 대각선 낙상, 서서히 주저앉는 동작, 일부 가려짐 상황에서 실패

#### 2단계: LSTM 시퀀스 분류 (학습 완료, 적용 예정)
- 데이터: **UR Fall Detection Dataset** (낙상 30 + ADL 15 시퀀스)
- 입력: 15프레임 × 132 feature (33 landmark × 4 value)
- 구조: LSTM(hidden=64, layers=2, dropout=0.3) → Linear(64→32) → Linear(32→2)
- 학습: 40 epoch, class weight, stride=3 (sliding window)
- 결과: val_acc/precision/recall 모두 1.0 — **소규모 검증 세트 과적합 의심**, 추가 데이터셋 (FDD, UP-Fall, TST) 필요
- 기대 효과: 순간 자세가 아닌 "넘어지는 과정" 자체를 인식 → 주저앉기·앉다 미끄러짐도 포착

### 6.2 복약 감지 — 객체 감지 기반 판정

#### 파이프라인
1. 보호자가 복약 스케줄 등록 (예: 매일 08:00 혈압약 1정)
2. 스케줄 시간 ±N분 동안 카메라가 해당 환자 인식 중
3. AI 서버가 프레임마다 YOLOv8 추론 → 약 객체 bbox 추출
4. 연속 프레임 누적 → 3프레임 이상이면 "의심" → 추가 단계에서 "완료"로 전환
5. 스케줄 종료까지 감지 없으면 "누락" → 보호자 FCM

#### 모델 선택 과정
- **초기**: `seblful/pills-detection` (ONNX, capsules/tablets 2클래스, mAP 91.7%)
- **중간**: YOLOv8n 베이스로 한국 약 41종 파인튜닝 → mAP50 0.34 (데이터 부족)
- **운영**: 3단계 폴백 체인으로 신뢰도 확보

#### 학습 환경
- RTX 4070 Ti, 40 epoch, batch=32, imgsz=640, amp=False (RTX 40 계열 호환성)
- EarlyStopping patience=10

#### 한계와 개선 방향
- 클래스당 평균 36장 → 최소 100장 이상 수집 시 mAP50 0.7+ 기대
- **손-입 이동 동작 분석** 미구현: 현재는 "약이 보인다"까지만 판정, "실제 먹었다"는 판정 불가 → MediaPipe Hands + 입 근접 좌표 비교 추가 필요

### 6.3 실시간 카메라 스트리밍 — MJPEG over HTTP

#### 왜 WebRTC가 아닌 MJPEG인가
- WebRTC: SDP 협상, STUN/TURN, ICE → 구현 복잡도 높음
- MJPEG: `<img src="/stream/:id">` 한 줄로 동작, 브라우저 네이티브 지원
- 트레이드오프: 대역폭 큼 / 지연 중간 (1~2초)
- 학생 프로젝트 범위에서는 **구현 단순성** 우선 → MJPEG 채택
- 필요 시 WebRTC로 전환 가능한 구조로 추상화 (`streaming/mjpeg.py` 모듈 분리)

#### 프론트엔드 캡처 루프
- `navigator.mediaDevices.getUserMedia`로 웹캠 획득
- `setInterval`로 **2 fps** 캡처 → canvas → base64 JPEG (품질 0.7)
- `/detect/live` POST → 응답(medication bbox + fall 상태)을 overlay canvas에 그림
- `inflightRef`로 중복 요청 방지

### 6.4 프론트엔드 목업 API 패턴 — DB 없이 UI 개발

```js
const useMock = async (fn, fallback) => {
  try { return await fn(); }
  catch { return fallback; }
};

export const api = {
  getPatients: () => useMock(() => request('GET', '/patients'), MOCK.patients),
  // ...
};
```

- 백엔드/DB가 안 떠 있어도 UI 시연 가능 → **시연회·발표 리허설에서 결정적 이점**
- 실제 API가 동작하면 자동으로 실데이터 사용 → 환경 전환 코드 변경 불필요
- 학생 팀에서 프론트/백엔드 병렬 개발할 때 대기 시간 제로

### 6.5 DB 설계 — Prisma 스키마 한 장으로 정규화

```
User (보호자)
 └── Patient (환자)
      ├── Camera (카메라 N개)
      ├── Medication (복약 일정 N개)
      │    └── MedicationLog (복약 기록 N개)
      ├── DetectionLog (감지 이력 N개)
      └── Notification (알림 N개)
```

- `Medication` 요일 반복은 CSV 문자열 (`"MON,TUE,WED,..."`) — 쿼리 단순성 우선
- `DetectionLog.type` 은 enum (`FALL`, `MEDICATION`, `NORMAL`)로 확장 여지 확보
- Prisma 마이그레이션으로 팀원 간 스키마 합의 비용 감소
